Termine & Intervalle – einfacher gestalten

Mehrere Beispiele, wie man Design und Termin-Felder sowie die vielen Arten elegant und unkompliziert lösen kann. Nur Darstellung (HTML), kein App-Code.

Beispiel 1: Eine Liste, ein Einstieg

Alle Termine in einer Liste. Ein Button „Termin hinzufügen“. Beim Tippen nur eine Frage: „Wiederkehrend oder einmalig?“

Termine · Muster GmbH
Fr, 14.02.EinmaligA + L
Mo, 17.02.RegelA
Di, 25.02.EinmaligA
Di, 04.03.EinmaligL

Dialog nach Klick: Zwei Optionen: „Wiederkehrend“ (dann: wie oft? wöchentlich / 14 Tage / monatlich) oder „Einmalig“ (dann: Datum + A oder A+L). Keine Unterscheidung Kunden-Termin vs. Ausnahme in der UI – optional ein Häkchen „Vom Kunden gewünscht“.

Beispiel 2: Nach „Wann“ gruppieren, nicht nach Typ

Nutzer denkt in Zeit: diese Woche, nächste Woche. Nicht in „Kunden-Termin“ vs. „Ausnahme“. Eine Liste pro Zeitraum, ein einheitliches Format.

Termine · Muster GmbH
Diese Woche
Fr 14.02.A + L
Mo 17.02.A
Nächste Woche
Di 25.02.A
Di 04.03.L

Keine Tags „Regel“ / „Ausnahme“ nötig – nur Datum und A/L. Art (wiederkehrend/einmalig) beim Bearbeiten sichtbar, wenn man den Termin antippt.

Beispiel 3: Nur zwei Blöcke – Regel vs. Einmalig

Statt fünf Bereiche (Nächster Termin, Kunden-Termine, Ausnahme, Regeln, Tour) nur zwei: „Regeltermine“ und „Einmalige Termine“. Gleiche Zeilen-Optik in beiden.

Termine · Muster GmbH
Regeltermine
Mo, 17.02.Wöchentlich · A
Mo, 24.02.Wöchentlich · A
Einmalige Termine
Fr, 14.02.A + L
Di, 25.02.A
Di, 04.03.L

„Regeltermine“ = aus Intervall/Wochentag/Liste (berechnet). „Einmalige Termine“ = Kunden-Termine + Ausnahme in einer Liste. Ein Einstieg „Termin hinzufügen“ mit Auswahl: Regel (wöchentlich/14 Tage/montag …) oder Einmalig (Datum + A/L).

Beispiel 4: Drei Karten beim Hinzufügen

Ein Dialog „Neuer Termin“: Nutzer wählt nur zwischen drei klaren Karten. Keine 8 Typen – nur drei Handlungen.

Neuer Termin
Schritt 1: Art wählen
Wiederkehrend Einmalig Ausnahme
Wiederkehrend → Wie oft?
Wöchentlich Alle 2 Wochen Monatlich

Dann: Startdatum + A-Tag (oder A+L). Fertig.

Vereinfachung der Arten: „Wöchentlich“ = bisher WEEKLY; „Alle 2 Wochen“ = intervallTage 14; „Monatlich“ = 1. Mo / 2. Di / … (MONTHLY_WEEKDAY). Ad-hoc/Unregelmäßig = „Einmalig“. Ausnahme = Sonderfall mit einem Häkchen oder eigener Karte. So hat der Nutzer nur 3 Entscheidungen, die App mappt intern auf die bestehenden Typen.

Beispiel 5: Kalender-Tag als Einstieg

Hauptansicht: Ein Tag (z. B. heute oder gewählt). Alle Termine an diesem Tag in einer Liste – egal ob Regel, Kunden-Termin oder Ausnahme. „Termin hinzufügen“ = „An diesem Tag einen Termin anlegen“.

Montag, 17.02.2026
Muster GmbHA
Andere FirmaA + L

Nutzer wählt Datum (Kalender/Pfeile). Sieht nur: Wer hat wann A/L. Hinzufügen = „Für welchen Kunden?“ → dann A oder A+L. Art (wiederkehrend/einmalig) kann optional im zweiten Schritt gefragt werden.

Beispiel 6: „Einmalige Termine“ ohne Trennung

Kunden-Termin und Ausnahme-Termin sind technisch gleich (Datum + A/L). In der UI: eine Liste „Einmalige Termine“. Optional pro Eintrag ein kleines Icon oder Text „vom Kunden“ / „Ausnahme“ – oder ganz weglassen und nur beim Anlegen fragen: „Vom Kunden gewünscht?“ Ja/Nein.

Einmalige Termine
Fr, 14.02.A + L
Di, 25.02.A
Di, 04.03.L

Im Backend bleiben zwei Listen (kundenTermine, ausnahmeTermine) oder eine mit Feld „quelle“. In der Anzeige: eine Liste, ein Format. Weniger Begriffe für den Nutzer.

Kurzvergleich der Beispiele

Beispiel Kernidee
1Eine Liste, ein Button; nur „Wiederkehrend / Einmalig“
2Gruppierung nach Zeit (diese / nächste Woche), keine Typ-Labels
3Zwei Blöcke: Regeltermine + Einmalige Termine, gleiche Zeilen-Optik
4Drei Karten beim Hinzufügen (Wiederkehrend / Einmalig / Ausnahme), dann „Wie oft?“
5Tag als Einstieg: alle Termine an einem Tag, hinzufügen = „An diesem Tag“
6Kunden-Termin + Ausnahme als eine Liste „Einmalige Termine“, optional Quelle verstecken

Alle Beispiele zielen darauf ab: weniger Begriffe, weniger Einstiege, ein einheitliches Darstellungsmuster. Die 8 technischen Arten (Regelmäßig, Monatlich, Wochentag, …) bleiben im Backend; in der UI werden sie auf 2–3 klare Handlungen reduziert.
